## Context

A GOOG cash-secured put executed on 2026-04-28 despite Google reporting earnings on 2026-04-29 (amc) — exactly the scenario the FC-007 earnings calendar was meant to prevent. Investigation showed the `EarningsCalendarService` exists and works (Finnhub returns the correct date for GOOGL — verified in `docs/plans/fc-007.md:24`), and is correctly used as a gate in `CallRoller.should_roll()` (call_roller.py:78-81). It is **not** wired into `PutSeller.find_put_opportunity()` or `CallSeller.evaluate_covered_call_opportunity()`. The FC-007 execution note acknowledged this gap explicitly: *"Integration with PutSeller/CallSeller log enrichment deferred to when those components are next modified."* That deferral has now produced a real production trade in front of earnings.

A holistic gate audit, prompted by the incident, surfaced more decay than the missing earnings hookup alone. **Three configured/declared gate elements are dead:**

1. `config.earnings_blackout_days` (config.py:570; settings.yaml:212) — defined but never read.
2. `config.earnings_avoidance_days` (config.py:441-443) — defined but never read; predates FC-007.
3. `RiskManager.validate_new_position()` (risk_manager.py:23) — defined but never invoked. Only `validate_roll()` is wired (from `CallRoller` only).

This violates `docs/CLAUDE.md` "Wheel Strategy Symmetry Principle": gates that apply to one phase of the wheel must apply to the other. Earnings is the most concrete violation; the audit doc shipped in this FC will catalogue the rest so the symmetry can be reasoned about at a glance.

## Goals

- Restore earnings symmetry: the Finnhub-based earnings blackout fires for **all three** trade-initiating paths (sell put, sell covered call, roll covered call).
- Activate the configured-but-dead `earnings.blackout_days` knob so it actually drives behavior.
- Change `EarningsCalendarService.is_earnings_within_n_days()` from fail-open to fail-closed — when Finnhub fails, all three paths skip the trade. (User-confirmed scope.)
- Publish `docs/gates.md` — a single source-of-truth audit of every gate across put-sell, call-write, call-roll, and orchestration phases, with file/line references and test-coverage state. Anchored in the Phase-1 inventory completed for this FC.
- Register sibling FC entries (FC-014, FC-015, FC-016) for asymmetries discovered in the audit but out of scope here.
- Remove dead config (`earnings_avoidance_days`).

## Non-goals

- Wiring `RiskManager.validate_new_position()` into the sellers (deferred → FC-014). It's been dead since at least FC-006 and fixing it cleanly requires more thought about portfolio-level checks (max exposure per ticker, min cash reserve) than belongs in a gate-symmetry FC.
- Centralizing `_entry_times` hold-period state (deferred → FC-015). Cold-start survival is the same class of bug as FC-009; group with that work.
- Adding test coverage to orchestration STAGES 3/4/5/6/9 and account-level gates (deferred → FC-016).
- Per-symbol earnings blackout (e.g., GOOGL=3, SOFI=2). Single global value `earnings.blackout_days` is sufficient given the universe size.
- Earnings-aware position closure logic (e.g., closing existing puts ahead of earnings). The wheel is designed to take assignment; closing pre-earnings would conflict with FC-010 philosophy.
- BigQuery view rewrites. New event types are additive; existing views continue to function.

## Proposed approach

### 1. Wire earnings gate into PutSeller and CallSeller

**`src/strategy/put_seller.py`** — add gate at top of `find_put_opportunity()` after the wheel-state check (around line 56):

```python
# Gate: Earnings blackout (FC-013 — symmetry with CallRoller)
if self.earnings_calendar:
    if self.earnings_calendar.is_earnings_within_n_days(
            symbol, self.config.earnings_blackout_days):
        proximity = self.earnings_calendar.get_earnings_proximity(symbol)
        log_trade_event(
            logger, event_type="put_blocked_earnings_blackout",
            symbol=symbol, strategy="sell_put", success=False,
            skip_reason="earnings_blackout", gate_failed="earnings_blackout",
            **proximity,
        )
        return None
```

Add `earnings_calendar` parameter to `PutSeller.__init__()` (default `None` for backward compat in tests).

**`src/strategy/call_seller.py`** — mirror in `evaluate_covered_call_opportunity()` after the wheel-state check (around call_seller.py:159-172). Same shape, event type `call_blocked_earnings_blackout`.

### 2. Inject `EarningsCalendarService` from WheelEngine

Today `EarningsCalendarService` is constructed only inside `_run_rolling_cycle` at wheel_engine.py:1357-1363. Hoist construction so the put/call cycle paths see it too.

- In `WheelEngine.__init__()`: construct `self.earnings_calendar = EarningsCalendarService(self.config) if self.config.earnings_enabled else None` once.
- Reuse the same instance for `PutSeller`, `CallSeller`, `CallRoller` (the cache benefits all three).
- Pass to `PutSeller`/`CallSeller` at instantiation.
- `_run_rolling_cycle` reads from `self.earnings_calendar` instead of constructing its own.

### 3. Flip fail-open → fail-closed in `EarningsCalendarService`

**`src/api/earnings_calendar.py`** — change `is_earnings_within_n_days()` and the proximity-fetch error paths to return `True` (block) on API errors instead of `False` (allow). Keep the 1-hour failure cache to avoid hammering a broken endpoint, and keep the `earnings_fetch_failed` error event with `recoverable=True`.

This is a behavior change for the existing roller path. Surfaced as a risk in Rollout. `earnings.enabled: false` remains the rollback lever.

### 4. Activate the live config knob, drop the dead one

- `config/settings.yaml`: keep `earnings.blackout_days: 2` (now actually consumed). Remove the `earnings_avoidance_days` entry under `risk.gap_risk_controls`.
- `src/utils/config.py`: keep `earnings_blackout_days` property (now consumed). Delete the dead `earnings_avoidance_days` property at lines 441-443.
- `tests/conftest.py`: drop `config.earnings_avoidance_days = 5` (line 141).

`rolling_earnings_blackout_days` (settings.yaml:202, config.py:529) stays unchanged — roller keeps its own knob so it can be tuned independently if needed.

### 5. Publish `docs/gates.md` audit doc

A canonical inventory keyed by **strategy phase × gate type**, derived from the Phase-1 audit. Sections:

1. Pre-trade gates per phase (put-sell, call-write, call-roll) with file:line, config keys, log events, test status.
2. Universe / orchestration gates (`wheel_engine` STAGES 1–9, `gap_detector`, `market_data` filters).
3. Account/execution gates.
4. Cross-strategy comparison table — flags asymmetries the way the Phase-1 audit did.
5. Known dead/disconnected entries (with FC pointers for cleanup).

This is the doc reviewers reach for when adding a new gate — answers "where does this belong, what's the symmetry obligation, where do I log it."

### 6. New log events

In `src/utils/logging_events.py` (or via existing `log_trade_event` if event_type is free-form):

- `put_blocked_earnings_blackout` — symbol, strategy=`sell_put`, success=False, skip_reason=`earnings_blackout`, gate_failed=`earnings_blackout`, plus `next_earnings_date`, `days_until`, `earnings_hour` enrichment.
- `call_blocked_earnings_blackout` — symmetric, strategy=`sell_call`.
- `earnings_fetch_failed_blocked_trade` — emitted from the fail-closed path so we can distinguish "blocked by real earnings" vs "blocked by Finnhub outage" in BQ.

### 7. Tests

- **New** `tests/test_earnings_calendar.py` (was a missing FC-007 deliverable per fc-007.md:152). Cover: cache hit, cache TTL expiry, Finnhub error → True (fail-closed), `is_earnings_within_n_days` boundary cases (0, 1, 2, 3 days), `get_earnings_proximity` shape.
- **Extend** `tests/test_put_seller.py` and `tests/test_call_seller.py`: gate fires inside blackout, gate allows outside blackout, gate blocks on Finnhub failure, gate is no-op when `earnings.enabled: false`, log event emitted with correct enrichment.
- **Update** `tests/test_call_roller.py`: existing test expects fail-open (`is_earnings_within_n_days.return_value = False`) — flip the relevant case to assert fail-closed behavior on API failure.

## Risks & mitigations

- **Fail-closed flip changes existing roller behavior.** Today a Finnhub outage allows rolls; after this FC it blocks them.
  - Mitigation: 1-hour failure cache already exists in `EarningsCalendarService`. `earnings.enabled: false` rolls back behavior atomically. Outages >1 hour are rare for Finnhub. Document the change in the deploy notes.

- **GOOGL is in our universe (settings.yaml:165) and reports 2026-04-29.** Deploying with fail-closed before market close on a near-earnings day means an unlucky Finnhub hiccup could halt activity at the worst time.
  - Mitigation: deploy outside market hours; verify `EarningsCalendarService._cache` is warm before enabling.

- **New event types may render as "Unknown" on the dashboard.** `mapEventToStatus()` in `dashboard/frontend/src/components/RecentTrades.tsx` (or similar) may not have entries for `*_blocked_earnings_blackout`.
  - Mitigation: cascading-impact check per `docs/CLAUDE.md`. Add mappings in the same PR if needed.

- **Stale 24-hour cache could miss a same-day earnings move.** A symbol with earnings announced midday wouldn't be picked up until the next refresh.
  - Mitigation: 24-hour TTL is already the FC-007 behavior; not a new risk. Earnings dates change ~quarterly, not midday.

- **Audit doc drifts out of date.** `docs/gates.md` is hand-maintained; new gates added in future PRs may not update it.
  - Mitigation: include a one-line PR-template item ("Updated docs/gates.md if this PR adds/changes a gate"). Long-term option (out of scope): a registry module the doc is generated from.

- **The earnings gate fires too broadly.** If `blackout_days: 2` and a put expires before earnings, we may skip a trade that wouldn't have been exposed.
  - Mitigation: 2-day blackout is the existing FC-007 + roller value; matches FC-007's risk model. If false-positives become a problem, tune the knob. Per-symbol overrides remain a non-goal.

2. After plan approval, implement on the same branch in a single PR.
3. Pre-deploy verification:
   - `pytest tests/test_earnings_calendar.py tests/test_put_seller.py tests/test_call_seller.py tests/test_call_roller.py -v`
   - Manual dry run: `python main.py --command run --dry-run` against a stock with known upcoming earnings (use GOOGL while the date is current).
   - Cascading-impact check on the dashboard event mapping.
4. Deploy to Cloud Run **outside market hours**.
5. Smoke check: trigger one strategy run; confirm at least one `*_blocked_earnings_blackout` event appears in BQ for a symbol within 2 days of earnings.
6. Monitor 1 trading week:
   - BQ: `SELECT event_type, COUNT(*) FROM trades WHERE event_type LIKE '%earnings_blackout%' GROUP BY 1`
   - Confirm zero put/call executions on stocks within 2 calendar days of earnings (cross-reference Finnhub).
   - Confirm `earnings_fetch_failed` events stay at baseline (no Finnhub regression from increased call volume).
7. Rollback lever: `earnings.enabled: false` (atomic, no code revert needed).

## Open questions

- Should `docs/gates.md` ship inside FC-013 or as a doc-only follow-up? Current plan: ship together — the audit doc IS the deliverable that proves the holistic-evaluation step happened.
- Should the gate live in `WheelEngine._scan_for_opportunities` (upstream of the sellers, as FC-007's original proposal at fc-007.md:98 suggested) or in the sellers themselves (current plan)? Current plan favors the sellers because (a) it's testable in isolation per component, (b) it mirrors `CallRoller.should_roll` placement, (c) it preserves the symmetry boundary at the seller level so a future call to `find_put_opportunity` from a non-engine caller still gets the gate.
- Do we want a feature flag `earnings.gate_put_call: bool` separate from `earnings.enabled`, or is one master switch sufficient? Current plan: one switch. If the granular flag becomes useful, add later.
