## Context

Dashboard data today is produced at runtime by the bot via two paths:

1. **Structlog → Cloud Logging → BigQuery sink** into `options_wheel.trades` (event-driven, broad coverage of bot activity)
2. **Direct BigQuery inserts** via `AnalyticsWriter` (`src/data/analytics_writer.py`) (performance/analytics tables)

Runtime generation ties dashboard correctness to bot uptime and to our event schemas staying in sync with downstream queries. When the bot crashes after a fill but before the event emits, the dashboard loses the trade. Every strategy change risks a log-schema change that has to propagate through the pipeline.

Alpaca maintains authoritative records for a subset of what we log — executions, assignments, expirations, exercises, order history, portfolio/equity history, position snapshots. We already use `/v2/account/activities` for reconciliation in `src/api/alpaca_client.py`. Where Alpaca is authoritative, **querying Alpaca is preferable to logging our own events** — simpler, idempotent, survives bot crashes, backfillable from inception.

This plan's first phase is an **audit**, not an implementation. We cannot decide what to migrate until we know what Alpaca can faithfully reproduce. Only after the audit completes does this plan fill in its Proposed Approach, Rollout, and Verification sections.

## Goals

- Deliver an exhaustive **current-vs-future-state audit** classifying every dashboard-read field as Alpaca-replaceable, Alpaca-derivable, partial, or not covered.
- Migrate all **(a)** and **(b)** bucket items to Alpaca-sourced queries.
- Backfill migrated tables with historical Alpaca data (pending retention validation).
- Reduce the surface area of `AnalyticsWriter` and/or structlog events where Alpaca supplants them.
- Keep dashboard functionality at parity or better — no regression in displayed data.

## Non-goals

- Replacing scan decisions, filtered symbols, risk-manager rejections, gap-detector triggers, or any "why the bot acted" events. These have no Alpaca equivalent and stay on the structlog pipeline.
- Replacing error events, circuit-breaker trips, execution-timing metrics, dedup/attempt counts.
- Removing BigQuery as the dashboard's data store. BigQuery remains; only the *source* of certain tables changes.
- Fixing strategy bugs unrelated to data sourcing (FC-008, FC-009) — though the audit will document whether migrating obviates them.

### Audit procedure

1. **Inventory `AnalyticsWriter`** — enumerate every method in `src/data/analytics_writer.py`, every table it writes to, every field it populates.
2. **Inventory `logging_events.py`** — enumerate every event type in `src/utils/logging_events.py`, every field each event carries.
3. **Inventory dashboard reads** — enumerate every query in `dashboard/backend/services/bigquery.py` and related routers under `dashboard/backend/routers/`. For each query, list the fields it selects.
4. **Map dashboard reads back** to the `AnalyticsWriter` methods and/or structlog events that produce them. This is the "what data the dashboard actually depends on" set.
5. **For each depended-on field**, determine which Alpaca endpoint (if any) carries the same information:
   - `/v2/account/activities` — all activity types (FILL, OPASN, OPEXP, OPXRC, CSD, CSW, DIV, FEE, INT, …)
   - `/v2/orders` — order history with status, timestamps, limit/stop/fill prices
   - `/v2/account/portfolio/history` — equity curve, P&L over time
   - `/v2/positions` — current open positions
   - Others as discovered
6. **Classify** each field into bucket (a)/(b)/(c)/(d) and fill the table below.

## Phase 2: Implementation (scaffolded, details pending audit)

Tentative sequencing — to be confirmed by audit output:

1. **Pilot with one migrated slice** (likely fills, as the simplest and highest-confidence mapping). New BigQuery table `trades_from_activities` written by a new Cloud Scheduler job.
2. **Validate parity** — run Alpaca-sourced table and existing `AnalyticsWriter` table in parallel for ≥1 week. Compare row counts, sums, per-trade P&L.
3. **Cut dashboard reads over** one table at a time once parity is confirmed.
4. **Deprecate the corresponding `AnalyticsWriter` path** once no dashboard read depends on it.
5. **Extend to remaining migration-manifest items** in dependency order.
6. **Backfill** historical data into each migrated table, using activity ID as dedup key against any existing overlap.

Each step produces a commit; revertible in isolation.

## Critical files

- `src/api/alpaca_client.py` — `get_account_activities()` already exists; likely extend for `/v2/orders`, `/v2/account/portfolio/history` as audit dictates.
- `src/data/analytics_writer.py` — subject of the migration. Methods here are the primary inventory target.
- `src/utils/logging_events.py` — secondary inventory target. Most events here are expected to remain (bucket d), but audit confirms.
- `dashboard/backend/services/bigquery.py` — dashboard-side queries. Whichever tables they read today are the migration targets.
- `dashboard/backend/routers/*.py` — endpoints consuming the BigQuery service. May need type updates if schema changes.
- `deploy/` — new Cloud Scheduler job + Cloud Run endpoint (or extension of existing) to run the Alpaca pulls.

## Reuse

- `src/api/alpaca_client.py` `get_account_activities()` — already handles pagination, REST auth, error fallback. Extend rather than rewrite.
- `src/strategy/wheel_engine.py` `_reconcile_positions()` — pattern for Activities-based idempotency (dedup by activity ID) is established here. Copy the shape.
- Existing BigQuery dataset `options_wheel` — new tables land alongside existing ones.

## Risks & mitigations

- **Alpaca retention may be shorter than account inception.**
  - Mitigation: validate empirically in Phase 1 before committing backfill scope.
- **Schema divergence during parallel-run period** — Alpaca-sourced and `AnalyticsWriter`-sourced rows may disagree on edge cases (rounding, timezone, fill splits).
  - Mitigation: write a reconciliation query that runs daily during parallel-run and alerts on divergence > threshold.
- **Rate-limit exhaustion** during backfill.
  - Mitigation: throttle to `<150 req/min` (below the 200 cap), run backfill off-hours, page by date range.
- **Dedup key mismatch** when backfilling into existing tables that have `order_id` but not `activity_id`.
  - Mitigation: backfill into new tables, cut dashboard over after parity, then retire old tables. No in-place merge.
- **Enrichment loss** — earnings_proximity, gap status, DTE-at-close are bot-computed and not in Alpaca.
  - Mitigation: join at read-time from a separate enrichment table keyed by `order_id`, OR accept their loss on Alpaca-sourced rows where the dashboard doesn't require them (audit will show).
- **Scope creep** — the "migrate wherever authoritative" framing is tempting to over-apply.
  - Mitigation: the audit's bucket classification is the hard gate. Items not in (a)/(b) do not migrate in this plan.

## Rollout

- **Phase 1 (audit)** produces a doc update to this plan file. No code change. No rollout.
- **Phase 2 (per-slice migration)** uses parallel-run pattern for each slice:
  1. Deploy new Alpaca-sourced writer behind a config flag `alpaca_sourced_analytics.<slice>: false`.
  2. Enable in shadow mode — write to new table, do not read from it.
  3. Compare against existing data for ≥1 week.
  4. Cut dashboard reads over once parity confirmed.
  5. Disable the corresponding `AnalyticsWriter` method.
  6. Remove the `AnalyticsWriter` code in a follow-up commit after one more week of clean operation.
- **Rollback**: any slice can be rolled back by flipping the dashboard read back to the `AnalyticsWriter` table (retained until full retirement).

## Verification

- [ ] **Production monitoring for 14 days** post-cutover per slice: no increase in dashboard error rates, no user-reported discrepancies.
