## Context

Dashboard data today is produced at runtime by the bot via two paths:

1. **Structlog → Cloud Logging → BigQuery sink** into `options_wheel.trades` (event-driven, broad coverage of bot activity)
2. **Direct BigQuery inserts** via `AnalyticsWriter` (`src/data/analytics_writer.py`) (performance/analytics tables)

Runtime generation ties dashboard correctness to bot uptime and to our event schemas staying in sync with downstream queries. When the bot crashes after a fill but before the event emits, the dashboard loses the trade. Every strategy change risks a log-schema change that has to propagate through the pipeline.

Alpaca maintains authoritative records for a subset of what we log — executions, assignments, expirations, exercises, order history, portfolio/equity history, position snapshots. We already use `/v2/account/activities` for reconciliation in `src/api/alpaca_client.py`. Where Alpaca is authoritative, **querying Alpaca is preferable to logging our own events** — simpler, idempotent, survives bot crashes, backfillable from inception.

This plan's first phase is an **audit**, not an implementation. We cannot decide what to migrate until we know what Alpaca can faithfully reproduce. Only after the audit completes does this plan fill in its Proposed Approach, Rollout, and Verification sections.

## Goals

- Deliver an exhaustive **current-vs-future-state audit** classifying every dashboard-read field as Alpaca-replaceable, Alpaca-derivable, partial, or not covered.
- Migrate all **(a)** and **(b)** bucket items to Alpaca-sourced queries.
- Backfill migrated tables with historical Alpaca data (pending retention validation).
- Reduce the surface area of `AnalyticsWriter` and/or structlog events where Alpaca supplants them.
- Keep dashboard functionality at parity or better — no regression in displayed data.

## Non-goals

- Replacing scan decisions, filtered symbols, risk-manager rejections, gap-detector triggers, or any "why the bot acted" events. These have no Alpaca equivalent and stay on the structlog pipeline.
- Replacing error events, circuit-breaker trips, execution-timing metrics, dedup/attempt counts.
- Removing BigQuery as the dashboard's data store. BigQuery remains; only the *source* of certain tables changes.
- Fixing strategy bugs unrelated to data sourcing (FC-008, FC-009) — though the audit will document whether migrating obviates them.
- Migrating `call_rolls` — explicitly excluded. Table's primary value is bot-internal decision data (skip reasons, earnings proximity, roll counts) which has no Alpaca equivalent. Stays on structlog path.

### Key Alpaca endpoint findings (sourced from sample API responses)

- **FILL** is the per-execution activity (one row per leg of an order). Fields: `transaction_time`, `price`, `qty`, `side` (e.g., `sell_short`), `symbol` (OCC format like `AMD260501P00277500`), `leaves_qty`, `order_id`, `cum_qty`, `order_status`. This is the granular record for individual option trades.
- **OPTRD** is a **daily settlement rollup per underlying** (symbol = underlying stock, not OCC). Appears once per trading day where an underlying had option activity. Not a substitute for FILL; useful only for daily reconciliation totals.
- **OPASN** (assignment): `date`, `symbol` (OCC), `qty`, `net_amount=0`, `status=executed`, `group_id` (joins to same-group FILLs/OPTRDs).
- **OPEXP** (expiration): similar shape to OPASN, `description="Options Expiry"`.
- **Important:** `FILL` uses `transaction_time` (ISO 8601 with ms); `OPASN`/`OPEXP`/`OPTRD` use `date` (day-granular) + `created_at`. Schema consumers must handle both.

### Audit findings & implications

1. **FC-008 and FC-009 sidestep confirmed.** `trades` table bugs (mislabeled stop-losses, duplicate close logging) come from structlog events. If dashboard reads trades from Alpaca FILLs instead, these bugs become analytics-irrelevant — they remain as operational log noise but no longer corrupt dashboard numbers. FC-008/FC-009 can be downgraded in priority or closed without separate fixes, pending Phase 2 cutover.

2. **Historical backfill is unblocked.** Paper account retention reaches inception (2025-10-06, 1,206 activities). Any retention-based risk to the backfill plan is empirically resolved for this account.

3. **Bucket (d) is large.** ~60% of AnalyticsWriter tables and dashboard queries cannot be migrated — operational, scan, and filtering data has no Alpaca equivalent. The migration is therefore **partial by design**, not a full replacement of the logging pipeline. This confirms the FC's hybrid framing.

4. **OPTRD is a trap.** Earlier FC-012 open questions implied OPTRD might be a migration candidate. It is not — it's a daily settlement rollup with coarser granularity than FILL. Use FILL exclusively for per-trade data.

5. **The legacy Cloud Logging sink (`options_wheel_logs` dataset) is a third data path** beyond the two originally framed in the FC. Dashboard does not currently read from it. Phase 2 should confirm this and either retire the sink or narrow its filter.

6. **`AnalyticsWriter.scans` methods are unused.** `write_scan_result()` / `write_scan_results_batch()` are defined but never called. Either the filtering pipeline writes scans via structlog only, or this is a stub. Worth confirming before any `scans` cleanup.

## Phase 2: Implementation (confirmed by audit; pending approval)

Sequencing below is the migration manifest from the audit, with rollout mechanics for each slice.

### 2.1 Pilot: `trades_from_activities` (raw) + `trades_with_outcomes` (view)

**Design principle:** append-only raw table + read-time projection into trade-lifecycle state. This cleanly separates *what Alpaca said happened* (raw events) from *how we interpret trades for the dashboard* (projection). See Decision 7 resolution below for the reasoning.

**Scope:** migration manifest item #1 — `get_recent_trades`.

- New Cloud Run endpoint `POST /ingest-activities` triggered by Cloud Scheduler on a split schedule: **every 15 minutes during market hours (9:30am-4pm ET, weekdays)** and **hourly off-hours**. Two separate Cloud Scheduler jobs with complementary cron expressions.
- Endpoint pulls `/v2/account/activities?activity_types=FILL,OPASN,OPEXP&after={last_seen}` paginated by `page_token`.
- Writes normalized rows to new table `options_wheel.trades_from_activities`. **Append-only** — no UPDATE, no MERGE. Idempotency via `INSERT ... WHERE NOT EXISTS` on `activity_id`, or equivalently: read existing activity IDs first and only insert new rows.

**Raw table schema (`options_wheel.trades_from_activities`):**

All columns set on insert, never modified:
- `activity_id` (STRING, REQUIRED, idempotency key)
- `activity_type` (STRING: `FILL` / `OPASN` / `OPEXP`)
- `transaction_time` (TIMESTAMP, partition key — for FILL; for OPASN/OPEXP synthesized from `date` + `created_at`)
- `activity_date` (DATE, ET calendar date)
- `order_id`, `symbol` (OCC for options, ticker for stock), `underlying`, `side`, `qty`, `price`
- `leaves_qty`, `cum_qty`, `order_status` (FILL only; null for OPASN/OPEXP)
- `group_id` (OPASN/OPEXP join key back to originating order)
- Parsed from OCC symbol: `option_type` (`put`/`call`/null), `strike_price`, `expiration`, `dte_at_event`
- Computed: `premium_total = price * qty * 100` (for option FILLs)
- `raw_payload` (STRING, original JSON blob for debugging) — optional but cheap

Table is time-partitioned on `transaction_time` (DAY granularity). Clustered on `symbol` for lookup efficiency.

**Ingest write pattern (append-only):**

Per cycle:
1. Read `MAX(transaction_time)` from `trades_from_activities` for FILL rows, `MAX(activity_date)` for OPASN/OPEXP rows. These are the cursor positions.
2. Pull `/v2/account/activities` with `after={cursor}`, paginated by `page_token` until exhausted.
3. Filter out any activity whose `activity_id` already exists in the table (defensive dedup via `SELECT activity_id FROM trades_from_activities WHERE activity_id IN (...)`).
4. Insert remaining rows via `insert_rows_json()`.
5. Done. No updates. No orphan scan needed.

**Projection view (`options_wheel.trades_with_outcomes`):**

Dashboard reads this view, not the raw table. Defined as a SQL view over `trades_from_activities` that pairs opening FILLs to close-side activities:

```sql
CREATE OR REPLACE VIEW `options_wheel.trades_with_outcomes` AS
WITH opens AS (
  SELECT * FROM `options_wheel.trades_from_activities`
  WHERE activity_type = 'FILL'
    AND side IN ('sell_short', 'sell')
    AND option_type IS NOT NULL
),
closes AS (
  SELECT symbol, activity_type, activity_date, transaction_time, price AS close_price,
         qty AS close_qty, order_id AS close_order_id, activity_id AS close_activity_id
  FROM `options_wheel.trades_from_activities`
  WHERE (activity_type = 'FILL' AND side IN ('buy_to_close', 'buy'))
     OR activity_type IN ('OPASN', 'OPEXP')
)
SELECT
  o.*,
  -- Earliest matching close event per opening OCC symbol
  ARRAY_AGG(c ORDER BY c.transaction_time ASC LIMIT 1)[SAFE_OFFSET(0)] AS close_event,
  CASE
    WHEN c.close_activity_id IS NULL THEN 'open'
    WHEN c.activity_type = 'OPASN' AND o.option_type = 'put' THEN 'assignment'
    WHEN c.activity_type = 'OPASN' AND o.option_type = 'call' THEN 'called_away'
    WHEN c.activity_type = 'OPEXP' THEN 'expiration'
    WHEN c.activity_type = 'FILL' THEN 'early_close'
  END AS outcome,
  -- realized_pnl computation per outcome type
  CASE
    WHEN c.close_activity_id IS NULL THEN NULL
    WHEN c.activity_type IN ('OPASN', 'OPEXP') THEN o.premium_total
    WHEN c.activity_type = 'FILL' THEN o.premium_total - (c.close_price * o.qty * 100)
  END AS realized_pnl,
  c.close_price AS outcome_price,
  c.transaction_time AS outcome_time,
  c.close_activity_id AS outcome_activity_id
FROM opens o
LEFT JOIN closes c
  ON c.symbol = o.symbol
  AND c.transaction_time > o.transaction_time
-- deduplicate to earliest close per open
QUALIFY ROW_NUMBER() OVER (PARTITION BY o.activity_id ORDER BY c.transaction_time) = 1
```

(Final SQL will be tuned during implementation; above is the shape.)

**Why a view, not a materialized view:** at current volume (~441 FILLs since inception, growing slowly) query-time projection is trivially fast. BQ scans one column-partitioned table. Start with a regular view; promote to materialized view only if query latency becomes noticeable. This defers a moving part.

**Eventual consistency:** OPASN/OPEXP activities arrive at ~8am ET the morning after the event. A trade that expired Friday gets its `outcome='expiration'` visible in the view as soon as Saturday morning's ingest cycle finishes.

**Cursor:** separate `MAX(...)` queries for FILL (on `transaction_time`) and for OPASN/OPEXP (on `activity_date`). Stored implicitly in the table itself; no separate cursor table needed.

### 2.7 Deprecation and cleanup

**Principle:** parallel-table strategy is a *temporary* state. Every v1 asset listed below must have a deletion date scheduled — not "we'll get to it." The worst outcome is leaving v1 tables, methods, and endpoints alive indefinitely, which defeats the purpose of the migration (a cleaner codebase and data surface).

**Trigger:** after all migrated queries are running on v2 sources for ≥14 days with zero divergence alerts.

**Deprecation checklist (must all land in a single follow-up PR, or explicit issue per item):**

Code to delete:
- [ ] `AnalyticsWriter.write_trade()` — no callers remain.
- [ ] `AnalyticsWriter.write_position_snapshot()` — both PORTFOLIO and non-PORTFOLIO paths.
- [ ] `AnalyticsWriter.write_scan_result()` and `write_scan_results_batch()` — dead on arrival; table is populated via Cloud Logging sink.
- [ ] The `scans` table schema definition in `AnalyticsWriter`.
- [ ] `bigquery.py` v1 methods (`get_recent_trades`, `get_premium_summary`, `get_premium_by_symbol`, `get_premium_by_day`, `get_pnl_by_symbol`, `get_portfolio_value_history`, `get_position_updates`) — callers now use v2 variants exclusively.
- [ ] Rename v2 methods back to their canonical names (`get_recent_trades_v2` → `get_recent_trades`) so the final code doesn't carry migration scars.
- [ ] `get_daily_stock_snapshots()` method, `/metrics/stock-snapshots` endpoint, and any frontend references (grep confirms none today — re-verify).
- [ ] The `dashboard.use_activities_source` feature flag — no longer needed after final cutover.

Data to remove:
- [ ] Drop `options_wheel.trades` table (v1).

- [ ] Drop `options_wheel.position_snapshots` table (both PORTFOLIO rows, migrated to `equity_history_from_alpaca`, and non-PORTFOLIO rows, dropped entirely).

Tables that stay:
- `options_wheel.errors`, `options_wheel.executions`, `options_wheel.scans`, `options_wheel.call_rolls` — all out of FC-012 scope.

**Follow-up scheduling:** once the ≥14-day parity window closes, schedule a dedicated cleanup PR within one sprint. Do not let the deprecation checklist drift more than 30 days past cutover. If any checklist item is blocked (e.g., an unexpected dashboard user surfaces), the blocker becomes a new FC and the remaining items still ship.

Each step produces a commit; each is independently revertible via the feature flag or by dropping the new table.

  - Mitigation: validate empirically in Phase 1 before committing backfill scope.
- **Schema divergence during parallel-run period** — Alpaca-sourced and `AnalyticsWriter`-sourced rows may disagree on edge cases (rounding, timezone, fill splits).
  - Mitigation: write a reconciliation query that runs daily during parallel-run and alerts on divergence > threshold.
- **Rate-limit exhaustion** during backfill.
  - Mitigation: throttle to `<150 req/min` (below the 200 cap), run backfill off-hours, page by date range.
- **Dedup key mismatch** when backfilling into existing tables that have `order_id` but not `activity_id`.
  - Mitigation: backfill into new tables, cut dashboard over after parity, then retire old tables. No in-place merge.
- **Enrichment loss** — earnings_proximity, gap status, DTE-at-close are bot-computed and not in Alpaca.
  - Mitigation: join at read-time from a separate enrichment table keyed by `order_id`, OR accept their loss on Alpaca-sourced rows where the dashboard doesn't require them (audit will show).
- **Scope creep** — the "migrate wherever authoritative" framing is tempting to over-apply.
  - Mitigation: the audit's bucket classification is the hard gate. Items not in (a)/(b) do not migrate in this plan.

## Rollout

- **Phase 1 (audit)** produces a doc update to this plan file. No code change. No rollout.
- **Phase 2 (per-slice migration)** uses parallel-run pattern for each slice:
  1. Deploy new Alpaca-sourced writer behind a config flag `alpaca_sourced_analytics.<slice>: false`.
  2. Enable in shadow mode — write to new table, do not read from it.
  3. Compare against existing data for ≥1 week.
  4. Cut dashboard reads over once parity confirmed.
  5. Disable the corresponding `AnalyticsWriter` method.
  6. Remove the `AnalyticsWriter` code in a follow-up commit after one more week of clean operation.
- **Rollback**: any slice can be rolled back by flipping the dashboard read back to the `AnalyticsWriter` table (retained until full retirement).

A SQL view `trades_with_outcomes` pairs opening FILLs to matching OPASN/OPEXP/close-FILL events at read time. Rationale: activities are genuinely event-shaped data; CQRS-style raw+projection is cleaner than MERGE, preserves debuggability ("why is this outcome wrong?" → query the raw activities), and the MV/view pattern handles growing-join concerns. If query latency ever becomes an issue at scale, promote the view to a materialized view — no ingest changes needed.
