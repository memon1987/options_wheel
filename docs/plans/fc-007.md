## Context

The codebase has an `earnings_avoidance_days` config value but no actual earnings date source. The gap detector checks overnight price gaps but has zero event awareness — it cannot distinguish an earnings gap from a random gap, and cannot prevent trading *into* upcoming earnings. This gap affects all three trading operations: put selling (biggest gap risk), call selling (volatility crush risk), and call rolling (FC-006 dependency).

Alpaca's corporate actions API covers dividends, splits, and mergers only — no earnings calendar exists, and there are no announced plans to add one.

Finnhub was selected after evaluating Alpaca, Yahoo Finance (yfinance), Alpha Vantage, and Polygon.io. Finnhub wins on free tier generosity (60 calls/min), per-symbol query support, official Python SDK, and SEC/exchange-sourced data quality.

## Goals

- Provide a shared earnings calendar service usable by all trading components
- Check "is symbol X reporting earnings within N days?" before any trade decision
- Enrich trade log events with earnings proximity data for BQ analysis
- Store Finnhub API key securely in Secret Manager

## Non-goals

- Historical earnings analysis or backtesting against past earnings
- Earnings surprise/estimate tracking (we only need dates)
- Real-time earnings announcement detection (pre-trade blackout is sufficient)
- Replacing gap detector (gap detector handles price gaps; this handles event risk)
- Persisting earnings data to BQ or GCS (in-memory cache + log enrichment is sufficient)

## Proposed approach

### 1. New module: `src/api/earnings_calendar.py`

```python
class EarningsCalendarService:
    def __init__(self, config: Config):
        self.client = finnhub.Client(api_key=config.finnhub_api_key)
        self._cache: Dict[str, Dict] = {}  # symbol → {date, hour, fetched_at}
        self.cache_ttl_hours = config.earnings_cache_ttl_hours  # default 24

    def is_earnings_within_n_days(self, symbol: str, n_days: int) -> bool
    def get_next_earnings_date(self, symbol: str) -> Optional[date]
    def get_earnings_proximity(self, symbol: str) -> Dict
        # Returns {next_date, days_until, hour} for log enrichment
    def _fetch_earnings(self, symbol: str) -> Optional[Dict]
        # Queries Finnhub, caches result, 90-day lookahead
    def _is_cache_valid(self, symbol: str) -> bool
```

**Storage:** In-memory cache only (dict keyed by symbol → earnings info). Refreshed once per symbol at first access each trading day. No persistent storage — Cloud Run instances are ephemeral, earnings dates change ~quarterly.

### 2. Config additions — `config/settings.yaml`

```yaml
earnings:
  enabled: true
  cache_ttl_hours: 24
  blackout_days: 2
  lookahead_days: 90
```

### 3. Config class — `src/utils/config.py`

- Add `finnhub_api_key` property (reads `FINNHUB_API_KEY` env var with Secret Manager fallback)
- Add to `_SECRET_MANAGER_MAP`: `"FINNHUB_API_KEY": "finnhub-api-key"`
- Add `earnings_cache_ttl_hours`, `earnings_blackout_days`, `earnings_lookahead_days` properties

### 4. Secret Manager

```bash
echo -n "<key>" | gcloud secrets create finnhub-api-key --data-file=-

# Update Cloud Run service to inject:
gcloud run services update options-wheel-strategy \
  --set-secrets FINNHUB_API_KEY=finnhub-api-key:latest
```

### 5. Integration into existing components

- `WheelEngine.__init__`: Instantiate `EarningsCalendarService(config)`
- `WheelEngine._scan_for_opportunities`: Before evaluating a symbol, call `earnings_calendar.is_earnings_within_n_days(symbol, config.earnings_blackout_days)`. If true, skip symbol and log.
- `PutSeller.execute_put_sale`: Add `next_earnings_date` and `days_to_earnings` to `log_trade_event` kwargs
- `CallSeller.execute_call_sale`: Same log enrichment
- `CallRoller` (FC-006): Use as gate in `should_roll()` decision criteria

### 6. Graceful degradation

If Finnhub is unreachable or returns an error:
- Log `earnings_fetch_failed` error event (`recoverable=True`)
- Return `is_earnings_within_n_days() → False` (allow trading, don't block on API failure)
- Cache the failure for 1 hour to avoid hammering a broken endpoint
- Rationale: Failing open is safer than blocking all trading because of a third-party API outage

## Risks & mitigations

- **Finnhub API outage blocks all trading**
  - Mitigation: Fail open — return False (allow trading) on API error, with 1-hour failure cache

- **Stale cache serves wrong data**
  - Mitigation: 24-hour TTL is conservative for data that changes ~quarterly. Worst case: miss a blackout by hours, not days.

- **Rate limiting on free tier**
  - Mitigation: 60 calls/min for ~14 symbols = negligible. Cache means 1 call per symbol per day.

- **`hour` field missing for some symbols**
  - Mitigation: Blackout logic uses calendar days, not timing — empty `hour` doesn't affect gating

## Rollout

1. Create `finnhub-api-key` secret in Secret Manager
2. Deploy with `earnings.enabled: true` (safe — only adds log fields, doesn't block trades until FC-006 uses it as a gate)
3. Verify log enrichment in BQ: `SELECT next_earnings_date, days_to_earnings FROM trades WHERE next_earnings_date IS NOT NULL`
4. After 1 week of clean data, FC-006 can depend on it for blackout gating
