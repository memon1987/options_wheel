# Release Notes — 2026-04-16

## Summary

Implemented the complete call rolling engine (FC-006) and its prerequisite earnings calendar service (FC-007). The system can now automatically roll expiring ITM covered calls to higher strikes on Fridays, with percentage-based debit tolerance and earnings blackout protection via Finnhub. Also fixed a pre-existing BigQuery view bug and established new plan-first workflow conventions.

## Changes

### New Features

- **Call Rolling Engine (FC-006):** Friday-only engine that buys-to-close expiring ITM short calls and sells-to-open new calls at higher strikes for next week. Sequential BTC -> STO execution with percentage-based debit tolerance (25% of original premium, 0.5% of notional backstop), 2-roll max per position, and partial fill tracking.
- **Earnings Calendar Service (FC-007):** Shared service using Finnhub API for earnings date lookups. Provides `is_earnings_within_n_days()` for trade gating and `get_earnings_proximity()` for log enrichment. In-memory cache with 24h TTL, graceful degradation (fails open on API errors).
- **POST /roll endpoint:** New Cloud Run endpoint with Friday guard, strategy_lock, and market hours check. Cloud Scheduler job fires at 3:30 PM ET every Friday.
- **call_rolls BigQuery view:** Dedicated view for roll event analytics, plus `/api/history/call-rolls` dashboard backend endpoint.
- **Per-position premium tracking:** WheelStateManager now stores `active_call_details` (option symbol, premium per contract, strike, contracts, sell date) populated by CallSeller after each execution. Enables debit tolerance calculations during rolling.
- **validate_roll() on RiskManager:** Roll-specific validation (roll-up only, cost basis protection, delta/DTE/premium checks) that skips irrelevant portfolio-level checks.

### Bug Fixes

- **position_updates BQ view filter mismatch:** View was filtering on `event_category='position_update'` but `log_position_update()` sets `event_category='position'`. Position update events were silently not reaching the view. Fixed to `'position'`.

## Key Decisions & Rationale

### FC-006 Plan Review (5 issues found and resolved)

1. **Earnings blackout source:** Original plan claimed "inherited from gap_detector" — gap_detector has zero earnings awareness. Resolved by creating FC-007 as a dependency. Evaluated Alpaca (no earnings API), Yahoo Finance (unreliable scraping), Alpha Vantage (25 calls/day too low), Polygon (paid only). Finnhub won on free tier, per-symbol query, and official SDK.

2. **Original premium data source:** Debit tolerance model requires knowing per-call premium, but WheelStateManager only stored aggregates. Extended state schema with `active_call_details` populated by CallSeller on each execution.

3. **Roll validation:** `validate_new_position()` validates new portfolio entries (position count, concentration, cash reserve) — irrelevant for replacements. Created dedicated `validate_roll()` that checks roll-specific concerns only.

4. **Partial fill handling:** Sequential limit orders can't guarantee atomicity. Decision: track and alert on partial STO fills, let Monday `/run` cycle sell calls on uncovered shares. Low severity — equivalent to HOLDING_STOCK phase.

5. **STO pricing:** Original plan had asymmetric aggression (5% BTC, 2% STO). Changed to symmetric 5% on both legs for thin Friday PM liquidity.

### Earnings data storage

Decided against persisting earnings dates to BQ or GCS. Earnings dates are public data queryable anytime from Finnhub. The analytical value is in the *decisions* (blocked/allowed trades), which are captured in existing trade log events enriched with `next_earnings_date` and `days_to_earnings` fields.

### Plan-first workflow updates

- Plan files renamed from `<slug>.md` to `fc-NNN.md` for traceability between FC entries and plans.
- Plan-First Development rules moved from repo-level `docs/CLAUDE.md` to parent `/Users/zmemon/CLAUDE.md` for cross-project persistence.
- Added post-execution plan updates rule: plan files must be updated with full GitHub PR URL, commit SHA, date, and notes after implementation.
- Added one-PR-per-FC rule for clean rollback.

## Discoveries

- **Alpaca has no earnings calendar API.** Corporate actions endpoint covers only dividends, splits, and mergers. Forum request from 2021 still unresolved. This was presented as a reusable component in the original FC-006 plan but doesn't exist.
- **`_parse_option_symbol` in CallSeller is private.** Public `parse_option_symbol()` exists in `src/utils/option_symbols.py` and should be used instead.
- **Existing scheduler jobs use OIDC auth** (not API key headers). The `/roll` endpoint follows the same pattern as `/run` and `/monitor` with `@require_api_key` decorator, but Cloud Scheduler authenticates via OIDC service account token.
