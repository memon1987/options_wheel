## Key Decisions & Rationale

**95% of ask for close limit orders:** Walked through a detailed GOOGL example. Considered mid-price (low fill probability, gamma risk while waiting), full ask (no savings), and 95% of ask (captures spread savings, high fill probability). The gamma risk near expiration makes delayed fills dangerous — saving $11 isn't worth risking a $180 profit reversal. Kept 95% for now; EVAL-001 will tune based on fill rate data.

**Activities API primary, position-diff fallback:** The Alpaca Activities API (`/v2/account/activities?activity_types=OPASN,OPEXP`) works with regular trading keys and returns 20 authoritative historical events. Position-diff was always an inference with false positive risk and cold-start blindness. Layered approach retains both.

**Dashboard rebuild vs fix:** Agent evaluation estimated fix at ~12 hours vs rebuild at ~10 hours. The frontend had no error boundaries, no loading timeouts, no retry logic — these are missing foundational patterns, not bugs. Rebuilding ensures every component handles loading/error/empty states from day one.

**BQ views with JSON_VALUE:** Date-sharded tables have inconsistent schemas across days (REPEATED vs STRING for same field). Direct `jsonPayload.field` references fail when ANY table in the wildcard range is missing that field. `JSON_VALUE(TO_JSON_STRING(jsonPayload), '$.field')` returns NULL instead of crashing — schema-tolerant across all tables.

**Not persisting share-ownership to _failed_symbols:** Intraday positions change (put assignment at 11am → shares available at 12pm). Only truly permanent failures (account_not_eligible, invalid_symbol) get persisted.

## Discoveries

- **IWM uncovered call root cause:** IWM had 100 shares + 1 existing short call (100 shares committed). Share verification checked total owned (100) vs required (100) and passed — but available was 0. The fix computes `available = owned - committed_to_existing_calls`.
- **Alpaca Activities API accessible with trading keys** — the SDK only exposes it on BrokerClient, but the REST API at `/v2/account/activities` works. 20 historical events (16 OPASN, 4 OPEXP) were always available.
- **6 completed wheel cycles identified:** AMD (-$3,750), AMZN (-$3,500), NVDA (+$250), UNH cycle 1 (-$1,750), UNH cycle 2 ($0), UNH cycle 3 ($0). Capital losses on 4 of 6 cycles — premium needs to exceed these for net positive.
- **BQ table architecture shifted:** Oct-Nov 2025 uses date-sharded tables; Nov 2025+ uses a single partitioned table. Wildcard `_*` covers both.
- **Dashboard spinning was purely frontend JS** — all 7 backend API endpoints returned 200 with valid data. The React app had no error boundaries and no loading timeouts.
