## Key Decisions & Rationale

**95% of ask for close limit orders (not mid-price, not full ask):**
Considered three options: (1) mid-price — best savings but low fill probability, especially near expiration where gamma risk means the position could reverse while waiting; (2) full ask — guaranteed fill like market order but no savings; (3) 95% of ask — captures ~$0.10-0.15 per contract savings while maintaining high fill probability. Chose #3 with ask as floor. After 2 weeks of fill data, EVAL-001 will determine if the multiplier should be adjusted.

**Activities API as primary, position-diff as fallback (not one or the other):**
The Activities API is authoritative (broker-recorded events with exact dates and prices) but could be unavailable (API outage, rate limit). The position-diff is unreliable as primary (false positives from manual trades, cold-start state loss) but catches cases the API might miss (e.g., if we fail to page through all activities). Layered approach: API first, diff as safety net.

**_failed_symbols doesn't persist share-ownership failures:**
Considered adding naked-call blocks to _failed_symbols (persists for the day). Rejected because intraday positions change — you could get assigned IWM shares at 11am and should be able to sell covered calls at 12pm. Only truly permanent failures (account_not_eligible, invalid_symbol) go in _failed_symbols.

**Kept run_strategy_cycle() despite being dead in production:**
It's the CLI entry point via main.py. Removing it would break `python main.py --command run`. Only removed its helper `_sync_wheel_state_with_positions()` which was a buggy duplicate of reconcile_positions().

**Removed evaluate_put_assignment() and evaluate_call_assignment_risk():**
These were dead code — defined in put_seller/call_seller but never called from any production path. They estimated assignment probability from moneyness, but the Activities API now gives us actual assignment records.

## Discoveries

- **8 of 14 configured symbols never trade:** AAPL (premiums too thin at our delta range), MSFT/QQQ/SPY (price above $400 max), F/PFE/KMI/VZ (premiums below $0.50 minimum). All pass initial filters but fail at stage 7 (no puts meet criteria). ~6,000 wasted API calls/month.
- **AMZN stopped trading after Feb 5** despite being the best ROI performer (123% annualized). Needs investigation — likely gap risk filter or position limit.
- **AMD gap-filtered 47% of the time** despite being the top premium generator. Gap risk thresholds may be too conservative for this symbol.
- **execution_engine routed ALL opportunities through put_seller** — call_seller.execute_call_sale() was completely unreachable from production. This caused the recurring "account not eligible for uncovered options" error.
- **merge_contextvars was missing** from cloud_run_server.py's setup_logging() — the correlation IDs we added were silently not appearing in any logs.
- **trades_executed BigQuery view** only surfaced 15 of 35+ available fields. Updated to include option_type, side, order_type, reason, market_value, unrealized_pl, profit_pct, etc.
- **All 202 early closes were market orders** — estimated $2k-5k in cumulative bid-ask slippage over 6 months.
- **handle_put/call_assignment() had a critical bug** — missing required `assignment_date` parameter meant every call silently failed with TypeError, caught by generic except. Assignment state transitions never executed in production.
- **Alpaca Activities API works with regular trading keys** despite the SDK only exposing it on BrokerClient. Direct REST call to `/v2/account/activities?activity_types=OPASN,OPEXP` returns full assignment/expiration history.
