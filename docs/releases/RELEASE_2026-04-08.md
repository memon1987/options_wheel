# Release Notes — 2026-04-08 (Session 4: Analytics Migration)

## Summary
Migrated the entire analytics/reporting data layer from fragile Cloud Logging → BigQuery log sink (auto-schema, date-sharded) to dedicated BigQuery tables with explicit code-defined schemas. This resolves the recurring schema conflicts that broke dashboard endpoints across sessions 2-3. All 12 dashboard endpoints now return 200. 2 commits, 12 files changed, +1,026/-146 lines.

## Changes

### New Features
- **AnalyticsWriter class** (`src/data/analytics_writer.py`) — manages 7 dedicated BQ tables with explicit schemas, time-partitioned by timestamp, graceful no-op if BQ unavailable
- **7 dedicated BigQuery tables** in `options_wheel` dataset:
  - `trades` (expanded with bid, ask, outcome, realized_pnl, filled_at)
  - `errors` (1,289 rows backfilled)
  - `executions` (forward-only)
  - `scans` (forward-only)
  - `wheel_cycles` (6 cycles backfilled)
  - `position_snapshots` (forward-only — portfolio value history)
  - `order_statuses` (forward-only — fill price tracking)
- **Analytics writes at 14 call sites** across cloud_run_server.py and wheel_engine.py
- **Position snapshot capture** — portfolio value, cash, buying power, and per-position P&L recorded at start of each /run cycle
- **Order status tracking** — filled/expired/canceled orders written to dedicated table from poll_order_statuses()
- **Backfill script** (`scripts/backfill_analytics.py`) — migrates historical data from log-sink to dedicated tables

### Bug Fixes
- **BQ dataset region mismatch** — `options_wheel` was created in `US` but log-sink data is in `us-central1`. Cross-region queries fail. Fixed both AnalyticsWriter and TradeJournal to use `us-central1`.
- **Reverted array logging** — `passed_symbols`, `rejected_symbols`, `failed_symbols`, `symbols` reverted from comma-joined strings back to Python lists. String logging created irreversible ARRAY<STRING> vs STRING schema conflicts in BQ daily tables.
- **Fixed `symbols` field collision** in options_scanner — `symbols=len(...)` (integer) renamed to `symbol_count=len(...)` to avoid type conflict with `symbols` array field in other events.

### Infrastructure
- **Dashboard backend fully repointed** — all 12 queries now hit `options_wheel.*` dedicated tables instead of `options_wheel_logs.*` log-sink views. No more JSON_VALUE, no more wildcard tables, no more schema conflicts.

## Key Decisions & Rationale

**Dedicated tables over log-sink views:** Google explicitly recommends against using Cloud Logging → BigQuery log sinks for analytics. The auto-detected schemas on date-sharded tables are inherently fragile — any field type change breaks wildcard queries permanently. We spent 3 sessions debugging schema conflicts. The TradeJournal pattern (direct writes with explicit schemas) has never had a schema issue.

**us-central1 region:** The log-sink dataset is in us-central1 (set by Cloud Logging). BigQuery can't do cross-region INSERT queries. The new analytics dataset must be co-located.

**NEVER convert structlog list kwargs to strings:** BQ auto-detects arrays as REPEATED fields. Converting to strings changes the column type, which is irrecoverable across daily tables. This rule is documented in session 3 memory.

**NEVER use JSON_VALUE(TO_JSON_STRING(jsonPayload)):** It serializes the entire STRUCT including REPEATED fields, which fails. Direct STRUCT access (`jsonPayload.field`) is required for Cloud Logging tables. But we no longer need either approach — the dashboard reads from dedicated tables, not log-sink views.

## Discoveries

- **BQ dataset location matters for INSERT queries** — creating a dataset in "US" (multi-region) while the source data is in "us-central1" causes 404 errors on cross-region queries. This is not well-documented by Google.
- **TradeJournal had already written 467 rows** to the trades table from previous sessions — this data survived the dataset recreation because the new dataset was empty (the old one was dropped and recreated).
