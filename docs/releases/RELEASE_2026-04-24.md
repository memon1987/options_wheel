### New Features

- **Alpaca activities → BigQuery ingest** (`src/data/activities_ingestor.py`): pulls FILL / OPASN / OPEXP activities into `options_wheel.trades_from_activities`. Append-only, idempotent by `activity_id`, paginated (100-page cap), cursor-driven by MIN(MAX(date)) across activity types.
- **Portfolio equity history ingest** (`src/data/portfolio_history_ingestor.py`): pulls `/v2/account/portfolio/history` daily EOD into `options_wheel.equity_history_from_alpaca`. Skips today's flapping row; idempotent by date.
- **Two new Cloud Run endpoints** (`deploy/cloud_run_server.py`): `POST /ingest-activities` and `POST /ingest-portfolio-history`, both protected by `@require_api_key` and invoked by Cloud Scheduler.
- **Two new BigQuery views** (`docs/bigquery/fc012_views.sql`):
  - `options_wheel.trades_with_outcomes` — projects raw activities into open trades paired with outcome (assignment / called_away / expiration / early_close / open). Dashboard reads this for all trade-execution queries.
  - `options_wheel.wheel_cycles_from_activities` — joins assigned puts to subsequent covered calls by underlying. Uses `LEAST(put_qty, call_qty)` for matched contracts to avoid over-counting on partial covers.
- **Three Cloud Scheduler jobs** (see `deploy/scheduler/fc012_activities_ingest.md`):
  - `activities-ingest-market-hours` — `*/15 9-16 * * 1-5 ET`
  - `activities-ingest-off-hours` — `0 0-8,17-23 * * * ET`
  - `portfolio-history-ingest-daily` — `30 16 * * 1-5 ET`

## Key Decisions & Rationale

### D1. Append-only raw + projection view vs. stateful MERGE (revised mid-plan)

Initially committed to "stateful ingest with pairing step UPDATEs outcome on opening rows when close arrives" to preserve one-row-per-trade semantics. Reversed to append-only raw + SQL view for outcome derivation after realizing:

1. The codebase has no existing mutation pattern — every BigQuery write is append-only. Introducing MERGE would set new precedent.
2. Alpaca activities are genuinely event-shaped data. Mapping them 1:1 to immutable rows is more honest than collapsing to state.
3. "Growing self-join cost" concern (the user's reason for preferring MERGE) is addressed by BQ's materialized-view option, not by compounding ingest complexity.
4. CQRS-style separation preserves debuggability — "why is this outcome wrong?" is answerable from the raw table. MERGE loses that history.

User challenge ("is append-only actually a design principle here?") was correct to force the re-evaluation. Accidental-convention-that-fits-this-case is a legitimate reason, not a lazy one.

### D2. Call rolls out of scope

Originally in the migration manifest. Dropped after realizing: (a) `call_rolls` is already on the structlog path, not `AnalyticsWriter` — no coupling to clean up; (b) most of its value is bot-internal decision data (skip_reason, earnings proximity, roll count) with no Alpaca equivalent; (c) migrating would add complexity (enrichment-table join) for marginal simplification. Revisit only if structlog-side data-quality bugs emerge on this table.

### D3. Drop `get_daily_stock_snapshots` entirely (not migrate)

User signaled that unrealized historical P&L per symbol is low-value because positions are closed regularly. Realized P&L is already covered by migrated queries. Grep confirmed zero frontend references to `/metrics/stock-snapshots`. Cleanest action was full removal, not Alpaca reconstruction.

### D4. Parallel tables + 30-day cleanup cap → dropped to single-PR cutover

Original plan called for v1/v2 parallel-run with ≥7 days of parity validation before feature-flag flip. User explicitly dropped the gate mid-plan ("dashboard is WIP, breakage is acceptable"). Consolidated phases 2.2-2.7 into one PR. Rollback is `git revert` instead of flag flip. Trade-off accepted because the dashboard wasn't a load-bearing production surface.

### D5. Outcome field via view (not MERGE)

See D1. The `trades_with_outcomes` view does the open/close pairing at read time via LEFT JOIN + ROW_NUMBER. Start with regular view; upgrade to materialized view only if query latency becomes noticeable (at current 465-row scale, it's instant).

### D6. Bookkeeping direct to main (not PR)

FC-008/009 re-evaluation and FC-012 status-to-Done updates landed direct to main as docs-only commits. Plan-first rule exempts "edits to docs/FUTURE_CONSIDERATIONS.md, docs/plans/*, and release notes themselves" — this qualifies.

## Discoveries

### Activity-type semantics

- **FILL** is the per-contract execution (symbol is OCC format, e.g. `AMD260501P00277500`). Has `transaction_time` (ISO 8601 with ms).
- **OPTRD** is a daily settlement rollup per underlying (symbol is ticker, e.g. `AMZN`). Appears once per day per underlying with option activity. **Not a FILL substitute** — coarser granularity.
- **OPASN** / **OPEXP** use day-granular `date` + `created_at` (not `transaction_time`). Arrive ~8am ET the morning *after* the event. Dashboard outcome is therefore eventually consistent for expirations/assignments.
- **Close-side FILLs report `side='buy'`, not `side='buy_to_close'`** — the view's CTE filter includes both for safety but the real data only has `buy`. Caught during FC-009 verification query.

### Retention + account shape

- Paper account has 1,206 activities from inception (2025-10-06 creation date).
- By type: FEE (720), FILL (441), OPASN (20), OPTRD (20), OPEXP (4), JNLC (1). Activity count ≠ trade count; FEEs are the majority.
- System is paper-only by design. No live-account retention validation needed.

### Deploy config surprises

- Cloud Build auto-triggers on push to `main` (took ~11 min for a full build — tests + build both containers + canary deploys + traffic shifts).
- `--set-env-vars` replaces all env vars. `--update-env-vars` merges. Using the former on Cloud Run deploys is a silent footgun.
- Cloud Run SA `roles/run.invoker` is per-user/SA (not `allUsers`) — required `gcloud auth print-identity-token` to hit endpoints during verification.

### Duplicate close-order incidents

## Known Issues

**FC-009 duplicate close orders — live bug, not yet fixed.** Verified against production data: 3 real duplicate-order incidents, all puts, ~$464 total spend on the redundant side. Mechanism: `_closed_today` in-memory dedup resets on Cloud Run cold starts. On every profit-target close of a put or call, if the close order hasn't filled before a cold start, the next monitor invocation re-evaluates the position and places a second close order. FC-010 reduced the *frequency* of vulnerable events (call stop-losses silenced) but did not patch the mechanism. Severity: operational-correctness, low cash impact at current scale; higher priority if volatility or position count grows. Fix options documented in FC-009.

**Historical `options_wheel.trades` / `wheel_cycles` / `position_snapshots` / `order_statuses` tables remain in BQ.** Dashboard no longer reads them; code no longer writes to `position_snapshots`. Other tables continue to get stale writes from TradeJournal and AnalyticsWriter (wheel cycles). Scheduled for manual `bq rm` drop on 2026-05-01 via remote routine `trig_01XYuhox9EhHQp26V1UeJAkp`.

**`cash` and `buying_power` are `NULL` on `/api/metrics/portfolio-chart` responses.** Alpaca's `portfolio/history` endpoint doesn't retain cash/buying_power historically. Frontend chart code that does arithmetic on these may need null guards.

**`wheel_engine._log_daily_stock_snapshots` still fires daily.** Nothing reads the log events anymore. Dead write-side work; harmless; left alone per Decision 4 (don't touch the Cloud Logging sink).
