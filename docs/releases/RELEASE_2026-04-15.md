  - **Plan references:** commit and PR messages include `Plan: docs/plans/<slug>.md`; FC entries move to "Completed" post-merge.

## Key Decisions & Rationale

**Precursor separate from todos.** FC entries are deliberately loose and unactionable. They exist so ideas don't get lost while research is pending, without pretending they're ready to execute.

**Plan-first gate at M/L, not all changes.** A hard rule on every change would block trivial fixes and discourage small improvements. The heuristic preserves velocity on typos/bug fixes while forcing structure on anything that changes behavior or risk.

**Async-friendly.** The user can iterate on FC entries and plans between sessions; Claude only starts coding once a plan is published or an override is given.

**Existing `docs/CLAUDE.md` over a new root CLAUDE.md.** The project already uses `docs/CLAUDE.md` as the canonical guidance doc. Adding the rule there keeps all project conventions in one place.

## Discoveries
- `PERFORMANCE_EVAL_CATALOG.md` serves a different purpose than a "future considerations" file: it catalogs recurring analytical queries/hypotheses with action thresholds, not open design questions. Keeping the two separate — evals drive parameter tuning, FC entries drive design work.
